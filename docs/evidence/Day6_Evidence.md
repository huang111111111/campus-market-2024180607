## 3. 状态判断顺序设计

四个列表页面统一采用以下 v-if 链：

```
<LoadingState v-if="loading" />
<ErrorState v-else-if="error" />
<EmptyState v-else-if="list.length === 0" />
<div v-else>正常列表</div>
```

**设计理由**：
- loading 优先级最高：请求中不应该显示错误或空状态
- error 次之：请求失败时应优先告知用户出错，而非显示"暂无数据"
- empty 第三：请求成功但无数据（或无搜索结果）时才有意义
- list 兜底：正常情况展示数据

相比优化前 `v-if="list.length === 0 && !loading"` 和 `v-if="list.length > 0"` 的并行判断方式，v-if/else-if 链更加清晰，且新增了 error 分支。

## 8. 遇到的问题与解决方法

### 问题 1：编辑 TradeView.vue 时模板被截断
- **现象**：使用 Edit 工具替换大段内容时，`</section>` 和 `</template>` 标签丢失，导致文件结构损坏
- **原因**：替换的 old_string 边界不精确，遗漏了模板闭合标签
- **解决**：使用 Write 工具完整重写整个文件，确保模板、脚本、样式三部分结构完整

### 问题 2：IDE 诊断显示 Variable not found 误报
- **现象**：编辑完成后 IDE 报告 keyword/error/loadItems 等变量不存在，但 vue-tsc 编译无错误
- **原因**：IDE 语言服务器在处理连续 Edit 时可能使用中间状态的缓存
- **解决**：运行 `npx vue-tsc --noEmit` 验证编译通过，确认是 IDE 缓存问题

### 问题 3：GroupBuyView 和 ErrandView 的旧 onMounted 未更新
- **现象**：第一次编辑时 script 中的 onMounted 仍使用旧的 try/catch 模式（无 loadItems 函数），但模板中已引用 loadItems
- **原因**：Old script section 在多次编辑中未完全替换
- **解决**：完整替换 script 中的 import + state + computed + functions + onMounted 全部内容

## 9. 今日反思

**交互优化**不是让页面"更好看"，而是让用户在任何情况下都知道发生了什么。今天为四个列表页面统一补充了三类状态反馈：加载中（用户知道正在请求数据）、加载失败（用户知道出了什么问题、可以做什么）、无数据（用户知道当前没有结果）。这三类反馈覆盖了用户使用过程中的主要场景，让"校园轻集市"从一个"能跑"的项目变成了一个"可用"的项目。

**加载状态**的意义在于管理用户预期。如果没有加载提示，用户看到空白页面会以为程序卡死了，可能频繁刷新或关闭页面。一个简单的旋转动画加上"正在加载XXX信息..."的文字，就让用户知道系统正在工作，只需稍等片刻。

**错误状态**的意义在于给用户出路。如果接口请求失败只写 console.error，用户看到的是空白页面，不知道发生了什么、也不知道该怎么办。今天实现的 ErrorState 组件不仅告诉用户"加载失败"，还提供了"重新加载"按钮，让用户有一个明确的下一步操作。

SearchBar 搜索组件让用户能够主动查找信息，而不是被动浏览。将搜索与分类筛选结合，用户可以更精确地定位目标内容。四个列表页面各有不同的搜索字段，但 SearchBar 组件本身保持通用，体现了"组件封装变化，页面定义差异"的设计思路。

Day6 的优化虽然不涉及新业务功能，但它决定了项目演示时的第一印象。一个加载顺畅、错误有提示、搜索可用的项目，远比一个功能虽多但空白报错的项目更能获得认可。
