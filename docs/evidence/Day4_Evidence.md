# Day4 Evidence - 发布表单与数据新增

## 1. 今日完成内容

本日完成了"校园轻集市"项目的发布表单设计、表单校验、数据提交与页面联动功能，具体包括：

- 在四个 API 模块（trade.ts、lostFound.ts、groupBuy.ts、errand.ts）中补充 **createXxx POST 方法**，支持向 JSON Server 新增数据
- 创建可复用组件 **FormField.vue**，统一表单项的标签、必填标识和错误提示
- 重写 **PublishView.vue**，实现完整的发布表单、表单校验、数据提交、成功反馈和页面跳转
- 支持四种发布类型的动态字段切换（二手交易、失物招领、拼单搭子、跑腿委托）
- 通过 JSON Server 的 POST 接口将新增数据写入 db.json，实现列表页联动显示
- 切换发布类型时自动重置表单，避免不同类型之间字段残留

## 2. 发布类型与字段设计

| 发布类型 | 对应数据集合 | 关键字段 | 设计理由 |
|---|---|---|---|
| 二手交易 | trades | title、category、price、condition、location、description | 价格和成色是二手交易核心决策信息；分类帮助筛选；地点影响交易可行性 |
| 失物招领 | lostFounds | title、type、itemName、location、eventTime、description | type 区分寻物/招领两种场景；itemName 明确物品本体；eventTime 记录发生时间 |
| 拼单搭子 | groupBuys | title、type、targetCount、deadline、location、description | targetCount 是拼单目标；deadline 决定拼单时效；type 覆盖拼餐、学习搭子等多种场景 |
| 跑腿委托 | errands | title、taskType、reward、from、to、deadline、description | from/to 描述路线；reward 是接单动力；taskType 区分取快递、代买、搬运等任务类型 |

**通用字段**：title、location、description 是所有业务类型都需要的，放在表单最前面，保证基础信息完整。

**字段与 db.json 一致性**：提交的字段名与 db.json 中已有数据字段名完全一致，确保新增数据能被列表页正确读取和渲染。

## 3. 表单校验规则

| 校验字段 | 适用类型 | 校验规则 | 错误提示 |
|---|---|---|---|
| title | 全部 | 不能为空 | 请输入标题 |
| location | 全部 | 不能为空 | 请输入地点 |
| description | 全部 | 不能为空 | 请输入描述 |
| category | 二手交易 | 不能为空 | 请输入商品分类 |
| price | 二手交易 | 必须大于 0 | 价格应大于 0 |
| condition | 二手交易 | 不能为空 | 请选择商品成色 |
| itemName | 失物招领 | 不能为空 | 请输入物品名称 |
| eventTime | 失物招领 | 不能为空 | 请选择发生时间 |
| groupType | 拼单搭子 | 不能为空 | 请选择拼单类型 |
| targetCount | 拼单搭子 | 不能少于 2 人 | 目标人数不能少于 2 人 |
| deadline | 拼单搭子/跑腿委托 | 不能为空 | 请选择截止时间 |
| taskType | 跑腿委托 | 不能为空 | 请选择任务类型 |
| reward | 跑腿委托 | 不能为负数 | 酬劳不能为负数 |
| from | 跑腿委托 | 不能为空 | 请输入取件地点 |
| to | 跑腿委托 | 不能为空 | 请输入送达地点 |

**校验设计原则**：
- 所有必填字段校验"不能为空"，确保提交数据完整
- 数值字段校验范围（价格 > 0、目标人数 ≥ 2、酬劳 ≥ 0），防止无意义数值
- 选择类字段校验"必须选择"，防止提交空选项
- 校验不通过的字段在对应 FormField 组件下方显示红色错误提示

## 7. 遇到的问题与解决方法

### 问题 1：JSON Server 自动生成的 ID 是字符串而非数字
- **现象**：POST 成功后返回的 `id` 是 `yxyyv1rx_XE` 这样的字符串，而 db.json 已有数据 id 是数字
- **原因**：JSON Server 1.x beta 版本默认生成 nanoid 风格的字符串 ID
- **解决**：使用 `Omit<TradeItem, 'id'>` 排除 id 字段让 JSON Server 自动生成，不影响页面渲染和功能

### 问题 2：无法连接 GitHub 推送代码
- **现象**：`git push` 报错 `Failed to connect to github.com port 443`
- **原因**：当前网络环境无法直接访问 GitHub
- **解决**：需要开启 VPN 或代理后再推送

## 8. 今日反思

**发布表单**是 Web 应用从"只读浏览"走向"用户交互"的关键环节。在"校园轻集市"中，用户不仅要能浏览二手商品、失物信息、拼单搭子和跑腿任务，还需要自己发布信息。设计发布表单不是简单堆几个输入框——它需要根据业务类型展示不同字段：二手交易需要价格和成色，失物招领需要区分寻物和招领，拼单搭子需要目标人数和截止时间，跑腿委托需要路线和酬劳。不同业务的字段选择直接反映各自的业务特征。

**表单校验**是保证数据质量的第一道防线。今天实现的校验规则虽然基础（必填、数值范围），但覆盖了最常见的用户输入错误。标签旁红色星号、输入框下方红色错误文字、校验不通过阻止提交——这三层机制构成清晰的用户反馈链路。好的表单校验不仅保护数据完整性，更重要的是给用户即时、明确的修改指引。

**数据新增**完成了从用户输入到数据持久化的完整链路。通过 Axios POST 请求，用户在表单中填写的内容被序列化为 JSON，发送到 JSON Server，写入 db.json，然后通过 GET 请求重新读取到列表页展示。这个"发布→存储→展示"的闭环让学生真正理解了 Web 应用的核心数据流。

发布表单、表单校验和数据新增三者密不可分。没有校验的数据新增不可靠，没有新增的发布表单无意义，校验规则的合理性又取决于表单字段的业务设计。Day4 为 Day5 的 Pinia 状态管理和后续用户交互优化提供了坚实的数据基础。
